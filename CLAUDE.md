# LuRavAntI — Project Guide for Claude Code

## What This Project Is

LuRavAntI is a business transformation consulting website. It is the public face of the company.
Primary domain: luravanti.com. Repo: github.com/lucadavdev/luravanti.

## Tech Stack (do not change without approval)

- **Frontend**: Next.js 15, App Router, TypeScript strict mode, Tailwind CSS v4
- **i18n**: next-intl — locales EN · IT · ES, prefix `/en/` `/it/` `/es/` always-on
- **Proxy**: Nginx 1.27 (HTTPS termination, security headers)
- **Database**: PostgreSQL 16 (currently empty — add tables only when a feature requires them)
- **Containerization**: Docker Compose — three services: `frontend`, `nginx`, `db`

## Brand & Design Tokens

- Background: `#050B22` (navy)
- Accent gradient: `#009CFF → #4E6CFF → #B83EFF` (left-to-right)
- Text primary: `#EAF0FF`
- Text muted: `#8B9CC8`
- Display font: Montserrat (font-display font-black for headings)
- Body font: Inter
- Utility class `grad-text` = brand gradient on text
- Dark, premium aesthetic. No light mode. No rounded-3xl+ cards.

## Content Rules

- **All copy lives in `frontend/src/i18n/messages/{en,it,es}.json`** — never hardcode strings in components.
- All 3 languages (EN/IT/ES) must be updated together. Never add a key to one file without adding it to all three.
- Tone: professional, confident, direct. No marketing fluff. No emojis in UI.

## SEO / GEO Rules (non-negotiable)

- Every page must have: unique `<title>`, unique `<meta description>`, canonical URL, hreflang alternates for all 3 locales.
- JSON-LD structured data (`Organization` in layout, `Service` on home page) must stay accurate and up to date.
- `/sitemap.xml` and `/robots.txt` are auto-generated by Next.js — do not create static versions.
- `/llms.txt` at `frontend/public/llms.txt` must be kept accurate as content or services change.
- One `<h1>` per page. Logical heading hierarchy (h1 → h2 → h3). No skipping levels.
- All `<img>` / `<Image>` elements must have descriptive `alt` text.

## LLM-Friendly Rules

- `/llms.txt` must describe every service and the company purpose accurately.
- Use semantic HTML: `<nav>`, `<main>`, `<section>`, `<article>`, `<footer>` — not generic `<div>` soup.
- Section IDs match nav anchors: `#approach`, `#services`, `#why`, `#process`, `#contact`.

## Architecture Rules

- **Never add a new npm dependency without justification.** Prefer Next.js built-ins.
- **Never store secrets in code.** All secrets via `.env` (gitignored). `.env.example` always updated.
- **`output: 'standalone'`** in `next.config.ts` — required for Docker multi-stage build.
- The Nginx container handles SSL. The Next.js container never terminates TLS.
- PostgreSQL password must come from environment variable — never hardcoded.

## File Conventions

- Components: PascalCase, named exports, one component per file.
- `frontend/src/components/sections/` — page sections only.
- `frontend/src/components/layout/` — Nav, Footer, shell components.
- `frontend/src/lib/` — non-UI utilities (schema, db, api helpers).
- `frontend/src/i18n/` — routing, navigation, messages.

## Git Conventions

- Branch from `main`, PR back to `main`.
- Commit format: `feat:`, `fix:`, `perf:`, `chore:`, `docs:`.
- Never commit: `.env`, `nginx/certs/`, `node_modules/`, `.next/`.
- Push after every logical commit.

## Dev Workflow

```bash
# Start dev environment
docker compose -f docker-compose.dev.yml up --build

# Or run frontend locally
cd frontend && npm run dev

# Verify all locales
open http://localhost:3000/en
open http://localhost:3000/it
open http://localhost:3000/es
```

## Adding a New Section

1. Add all string keys to `en.json`, `it.json`, `es.json` simultaneously.
2. Create `frontend/src/components/sections/NewSection.tsx`.
3. Import and mount in `frontend/src/app/[locale]/page.tsx`.
4. If the section introduces a new service, update `/llms.txt` and the `servicesSchema()` in `frontend/src/lib/schema.ts`.
5. Update `<meta description>` if page summary changes.

## Adding Backend Features

When a backend feature is needed:
- Add the table in `db/init.sql` with migration comments.
- Add API routes under `frontend/src/app/api/` (Next.js route handlers).
- Never expose DB credentials to the frontend.
- If a new service is needed (e.g., a background worker), add it to `docker-compose.yml` and `docker-compose.dev.yml`.
